# 六、项目复盘与总结

> 前五个主题回答了"做了什么"。这一章回答的是：做完之后，你真正学到了什么。从"项目完成"升到"经验沉淀"。

---

## 6.1 关键成果总结

### 这一小节要解决什么问题

用更抽象的方式总结项目的核心验证，不重复第五章的结果表。回答：这个项目最终验证了哪些核心判断？自研链路证明了什么？Qwen 迁移线证明了什么？为什么说它是完整闭环？

### 核心结论

**从零搭建完整训练链路是可行的。** MicroLM 的每一步——BPE tokenizer（从字节到 6400 词表）、TransformerLM（einsum Linear / RoPE / SwiGLU / RMSNorm）、自实现 AdamW + cosine scheduler + gradient clipping、assistant-only masked loss ——全部手写，没有依赖 HuggingFace 的模型层。这证明了对 Transformer 架构的理解可以落实到每一行代码上，而不只是调用 API。

**LoRA 在微型模型上同样有效。** 两个独立验证：MicroLM 上 0.83% 参数达到全参 SFT 85% 的生成质量；Qwen 上 0.14% 参数的 val_loss 降幅达 61.6%。参数效率不是大模型专利。

**方法论可以跨生态迁移。** 自研链路的配置组织方式（JSON config → resolved config）、日志格式（JSONL per-step）、训练循环结构（step loop → val → checkpoint）、报告模板——这些设计模式在迁移到 HF/PEFT/vLLM 工具栈时几乎原样复用。需要重新适配的是接口层（tokenizer / chat template / LoRA 框架），不是思维方式。

**聚焦比泛化更容易出硬指标。** 阶段 6 放弃通用聊天增强、聚焦 InstructIE 结构化抽取后，评测体系从人工打分变成了四项自动检测指标，部署决策从"感觉差不多"变成了 alias-strict 15.0% vs 7.5% 的数字对比。窄而深 > 宽而浅。

### 量化成果一览

| 维度 | 自研链路 (MicroLM) | 迁移链路 (Qwen) |
|------|-------------------|----------------|
| 模型规模 | 31.7M 参数 | 1.55B + 2.18M 可训练 |
| 训练闭环 | pretrain → SFT → LoRA 全部完成 | 数据构建 → LoRA → 评测 → 部署 |
| LoRA 效率 | 0.83% 参数，99.7% 存储节省 | 0.14% 参数，99.7% 存储节省 |
| 微调效果 | val_loss 差距 9%（vs 全参）| val_loss 降幅 61.6% |
| 推理加速 | KV Cache 平均 3.86x，最大 9.08x | — |
| 生成质量 | SFT 评分 +81%（vs pretrain）| Alias-Strict 为 base 2 倍 |
| 部署状态 | chat.py 交互式 REPL | vLLM OpenAI 兼容 API |

### 这一节的结论

这个项目的本质是一次 **end-to-end 的方法论验证**：从数据到 tokenizer 到模型到训练到微调到推理到评测到部署，每个环节都亲手实现并验证了可行性。双轨并行（自研 + 开源迁移）的设计让每条线的经验都可以作为另一条的对照系。

## 6.2 关键 Bug、踩坑与修复

### 这一小节要解决什么问题

记录项目中遇到的每一个关键 bug——不是简单罗列"报了什么错"，而是写清现象、根因、修复和教训。每一个 bug 都暴露了一个隐藏的设计假设，记录它们的价值在于理解为什么当初没发现——以及以后怎么避免同类问题。

> [!warning] 这些 bug 不是"代码写错了"那么简单
> 每一个都暴露了一个隐藏的设计假设。记录它们的价值不在于 bug 本身，而在于理解为什么当初没发现——以及以后怎么避免同类问题。

### Bug 清单

#### Bug 1 — Tokenizer 词表与模型 Embedding 不匹配

- **现象**：多轮对话第二轮时 `IndexError: index out of range`
- **根因**：tokenizer 训练时 vocab_size 设为 6400，但后续注册 special token（EOS 等）后词表实际大小变为 **6401**。EOS 的 token ID = 6400。而模型 embedding 层的 vocab_size 仍然是 **6400**——它是根据原始配置创建的，不知道 tokenizer 后来加了 special token
- **修复**：两层防御：(1) 仅在 eos_token_id < model.vocab_size 时才追加 EOS 标记；(2) 编码后裁剪所有 ≥ vocab_size 的 token ID。同时在 train_sft.py 中增加 embedding/lm_head 的 resize 逻辑
- **教训**：tokenizer 和模型是两个独立模块，各自有自己的"词表大小"概念。当中间插入 special token 注册操作时，两边不会自动同步。任何修改词表的操作都必须同时检查下游消费者

#### Bug 2 — SFT 推理格式与训练格式不一致

- **现象**：SFT checkpoint 的生成质量明显低于预期，尤其是多轮对话的第一条回复经常跑题
- **根因**：`_render_prompt()` 在 assistant 消息后缺少 EOS 标记。SFT 训练数据中每条 assistant 回复都以 EOS 结尾，模型学到的是"输出一段文字然后遇到 EOS 就停"。推理时如果 prompt 中没有这个 EOS 标记，模型的分布起点就和训练时不一致
- **修复**：在 assistant 消息渲染后统一追加 EOS + 换行
- **教训**：SFT 的训练-推理一致性不仅在于模型权重，还在于 prompt 格式的每一个字符。loss mask 保证梯度只流向 assistant 区间，但 assistant 区间的**边界定义**必须训练和推理完全一致。这类 bug 不会报错，只会静默地降低生成质量——最危险的那种

#### Bug 3 — Checkpoint 加载顺序导致 LoRA 权重丢失

- **现象**：加载 pretrain checkpoint 后再 apply LoRA，LoRA adaptor 的训练从随机初始化开始而非基于 pretrain 权重
- **根因**：`load_model_state()` 加载权重后，`apply_lora_to_model()` 替换 Linear 为 LoRALinear 时，新创建的 LoRALinear 从头初始化其内部 W 权重——但执行顺序反了：先 apply_lora（创建新的 LoRALinear 结构）再 load_state_dict（填入权重），如果 load 的逻辑只匹配原始 Linear 的 key 而不匹配 LoRALinear 内部的 `_orig_mod.` 前缀，权重就丢了
- **修复**：调整加载顺序，确保 load_state_dict 能正确处理已注入 LoRA 的模型状态字典（处理 `_orig_mod.` 前缀）
- **教训**：LoRA 的 state_dict 结构和原始模型不同（多了 A/B 矩阵，原始 W 变成 `_orig_mod.weight`）。任何涉及 checkpoint save/load 的代码都必须感知这种结构变化

#### Bug 4 — LoRA 参数设备不匹配

- **现象**：LoRA 训练开始后 device assert 失败或 loss 不下降
- **根因**：`LoRALinear.__init__` 创建 A/B 参数时未指定 device 参数，默认在 CPU 上创建。而模型其他参数已在 GPU 上。前向传播时出现跨设备计算错误
- **修复**：在创建 lora_A 和 lora_B 时显式指定 `device=original.weight.device`
- **教训**：PyTorch 中参数的 device 创建是隐式的。当自定义模块（如 LoRALinear）在 `__init__` 中创建新参数时，必须显式继承原始权重的 device——否则在多 GPU / mixed precision 场景下会出问题

#### Bug 5 — InstructIE 数据集的字段漂移

- **现象**：数据 pipeline Step 1 之前，直接在原始数据上做统计和过滤，结果发现大量样本字段对不上
- **根因**：InstructIE 的 train / valid / test 三个 split 字段命名不统一：train 用 `text`，valid/test 用 `input`；train 的 relation 含 `head_type`/`tail_type`，valid/test 不含；cate 名称有漂移："建筑结构" vs "建筑"
- **修复**：Step 1（`01_normalize.py`）作为 pipeline 第一步专门做字段标准化
- **教训**：拿到一个新数据集，第一件事永远不是"开始写 training loop"，而是 **profiling**——统计字段分布、检查 cross-split 一致性、抽样肉眼观察

#### Bug 6 — 验证集误用训练集

- **现象**：MicroLM LoRA smoke test 的 val_loss 异常低，正式训练时却高很多
- **根因**：配置文件中 valid_data_path 指向了训练集路径（895K 条）而非专用验证集（8.7K 条）。模型在"验证"时看到的是自己训练过的数据
- **修复**：修正配置路径，使用独立的验证集切分。eval 速度因此提升约 **100 倍**
- **教训**：val_loss 是唯一能告诉你"是否过拟合"的信号。如果这个信号本身是假的（因为验证集就是训练集），你就失去了所有的训练过程监控能力。配置文件中的路径应该经过一次性的系统性审查

#### Bug 7 — wandb mode 配置错误

- **现象**：启动训练后 wandb 报错中断
- **根因**：配置中 wandb mode 写成了无效值（如 "enabled" 或数字而非字符串）
- **修复**：修正为 "online" / "disabled"
- **教训**：训练工具链上的配置错误意味着每次实验都被打断。适合加一层 enum validation 或启动时校验

#### Bug 8 — Unicode Surrogate 字符崩溃

- **现象**：chat.py 多轮对话第二轮 `UnicodeEncodeError`
- **根因**：小模型生成的某些 token 序列解码后产生 Unicode surrogate 字符（U+D800–U+DFFF），存入对话历史后下一轮 `.encode("utf-8")` 时崩溃。这类 bug 只在多轮对话的第二轮及之后触发——问题出在"历史的累积"而非单次生成
- **修复**：新增 `_remove_surrogates()` 工具函数，每轮回复存入历史前自动清理
- **教训**：累积型 bug（历史状态逐步恶化）比一次性 bug 更难排查，因为单次运行正常。对任何涉及"历史""缓存""累积"的代码都要特别警惕

### 这一节的结论

8 个 bug 可以分为三类：
- **协议/格式类**（Bug 1, 2）：训练和推理之间的隐式假设不一致——最危险因为不会报错
- **状态管理类**（Bug 3, 4, 8）：模块间状态不同步——device 不匹配、state_dict 结构变化、历史累积
- **数据/配置类**（Bug 5, 6, 7）：外部输入的隐式假设被违反——字段漂移、路径错误、枚举值无效

共同教训：**任何跨模块的接口都必须有显式的契约定义**，不能依赖"应该是一样的"这种隐式假设。

## 6.3 方法论收获

### 这一小节要解决什么问题

从具体的技术实现中提炼出抽象层面的经验。这些经验不限于本项目，可以迁移到未来的 ML 工程实践中。回答：哪些做法被证明是对的？哪些原则值得坚持？

### 核心方法论

**配置驱动优于硬编码。** 项目中 17 个 JSON 配置文件覆盖全部阶段的超参数。好处不仅是"改参数不用改代码"，更重要的是每个实验都有可追溯的完整条件记录（`resolved_train_config.json` 固化了实际运行时的所有参数值，包括路径解析后的绝对路径）。这在对比实验和复现时省了大量时间。

**数据协议显式化很重要。** MicroLM 的 SFT 数据协议（sft.py）和 Qwen 的三段式格式（instruction + schema + input → JSON output）都是"先定义协议再写训练代码"。这让后续做数据治理、模板改造、LoRA 适配时不容易混乱——因为数据长什么样是写在文档里的，不是隐含在代码逻辑里的。

**先 smoke 再正式。** 每个阶段都遵循 smoke test → 正式训练的模式（pretrain smoke → full corpus, SFT smoke → baseline, Qwen LoRA smoke → 2000 步）。smoke 用 20 步 / 50 步 / 小数据跑通全链路，确认数据加载正常、loss 能下降、checkpoint 能保存后再投入正式计算。这避免了"跑了 10 小时发现 loss mask 写反了"的惨剧。

**评测要先于优化。** 阶段 3（固定 prompt 评测）在阶段 1（LoRA 实验）之后补齐，但它的价值应该在更早的阶段就被认识到——如果没有统一的评测基线，就无法判断后续改动是进步还是退步。建立评测体系的最佳时机是在第一次实验之前，而不是在做了很多实验之后。

**自研和开源生态各自的价值。** MicroLM 自研链路证明了"能从零搭起来"——这对理解原理至关重要。但 Parse%=0% 也暴露了小规模自研模型的天花板。Qwen 迁移线证明了两件事：(1) 同样的方法论在工业工具栈上确实能跑出更好的绝对效果；(2) 从自研迁移到开源的工作量主要在接口适配层，不在思维模式层。对于一个 LLM 工程师来说，两条线都需要掌握——自研让你理解原理，开源让你交付产品。

**双轨并行的组织方式值得保留。** 两条链路共享设计模式但技术栈完全独立，这意味着任何一个链路的经验都可以作为另一个的对照系。比如 MicroLM 的 KV Cache benchmark 测的是算法加速比，Qwen 的 vLLM benchmark 测的是系统吞吐——两者回答不同层面的问题，但共同构成了"推理优化"这个能力的完整图景。

### 这一节的结论

方法论层面的收获比具体技术点更有持久价值。配置驱动、协议显式化、smoke-first、评测先行——这些原则在任何 ML 工程项目中都适用，不限于 LLM 领域。它们共同指向一个核心理念：**工程化的 ML 项目不是一堆散落的脚本，而是一套有纪律的方法论**。

## 6.4 后续可扩展方向

### 这一小节要解决什么问题

项目完成后还能往哪里走。不需要太长，但要区分短期（可直接做）、中期（需额外工作但路径清晰）、长期（需重新规划）三个层次。

### 短期延伸（基于现有代码和数据可直接做的）

| 方向 | 做法 | 预期产出 |
|------|------|----------|
| 更大的 Qwen 基座 | 迁移至 Qwen2.5-7B-Instruct，复用相同 pipeline | 更强的结构化抽取基线，验证 scale-up 效果 |
| 多轮结构化对话 | 扩展 SFT 数据加入 multi-turn IE 任务 | 支持"逐步细化抽取"的交互场景 |
| Function Calling | 基于 schema-following 能力扩展为 tool use 格式 | 直接服务于 Agent 场景 |
| 量化部署 | INT8 / INT4 量化后的 vLLM benchmark | 降低部署硬件门槛，补充量化-精度权衡数据 |

### 中期延伸（需要额外工作但路径清晰）

| 方向 | 说明 |
|------|------|
| RAG + 结构化抽取结合 | 先检索相关文档，再用 qwen_lora 做 schema-guided 抽取，形成完整的问答管线 |
| 评测体系扩展 | 加入更多开源 benchmark（CMMLU / C-Eval 的中文子集），建立通用能力 + 结构化能力的二维评估 |
| 数据飞轮 | 将 vLLM 服务收集的真实请求和纠错反馈回流到 SFT 数据集，迭代微调 |

### 长期方向（需要重新规划）

| 方向 | 与当前项目的关系 |
|------|----------------|
| 多模态扩展 | 当前纯文本，可引入图像→结构化描述的任务变体 |
| Agent 框架集成 | 当前的 structured output 能力是 function calling / tool use 的前置基础 |
| 更大规模的数据工程 | 当前 InstructIE 单源 28.5K 条，可扩展到多源混合（DocRE / ACE / 自建标注）|

### 这一节的结论

短期延伸的风险最低——现有 pipeline 和代码可以直接复用。中期延伸需要额外的工程投入但不需要改变架构。长期方向可能需要重新评估项目定位。对于简历项目和面试展示而言，短期延伸（特别是 Function Calling 和量化部署）的投资回报率最高——它们直接扩展了项目的技术深度和工程完整性叙事。
