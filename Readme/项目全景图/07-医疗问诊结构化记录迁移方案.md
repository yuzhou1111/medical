# 七、医疗问诊结构化记录迁移方案

> 本文把当前项目的 Qwen LoRA + 结构化输出链路迁移到“医疗问诊结构化记录”垂直领域，并给出可直接复制给更强模型执行项目重构的提示词。目标不是让模型做诊断，而是把医生/患者对话、门诊问诊文本或病史描述稳定转写为可审计 JSON 记录。

---

## 7.1 迁移目标

当前项目已经具备：

- `scripts/01~06_*.py` 的结构化数据 pipeline；
- `scripts/train_qwen_lora.py` 的 Qwen2.5 LoRA 微调；
- `eval/prompts_instructie.json` 的结构化评测协议；
- `scripts/run_instructie_eval.py` 与 `scripts/check_structured_stability.py` 的 JSON 可解析率、schema strict、alias strict 评测。

医疗迁移的核心是把原来的“通用信息抽取 schema”替换为“问诊记录 schema”：

```text
医生/患者对话或病情描述
  -> 去标识化 + 质量过滤 + 任务派生
  -> chat-style SFT JSONL
  -> Qwen LoRA 微调
  -> 结构化问诊记录 JSON
  -> 离线评测 + vLLM 稳定性验证
```

**安全边界：**

1. 只做记录结构化，不做诊断裁决。
2. 不编造文本中没有的信息。
3. 对否认、未提及、不确定信息要显式区分。
4. 输出不得包含治疗建议、处方建议或替代医生判断的内容。
5. 训练数据必须去标识化，移除姓名、电话、身份证号、住址、医院号等敏感信息。

## 7.2 推荐输出 Schema

采用单根对象，便于现有评测脚本识别字段：

```json
{
  "问诊记录": {
    "主诉": "发热咳嗽2天",
    "现病史": "2天前受凉后出现发热...",
    "起病时间": "2天前",
    "症状": ["发热", "咳嗽"],
    "伴随症状": ["咽痛"],
    "否认症状": ["胸痛", "呼吸困难"],
    "既往史": "否认慢性病史",
    "用药史": "自行服用布洛芬一次",
    "过敏史": "青霉素过敏",
    "生命体征": {"体温": "38.5℃"},
    "风险信号": [],
    "初步分诊": "普通门诊",
    "建议就诊科室": "呼吸内科",
    "随访问题": ["是否有基础肺病史？"]
  }
}
```

字段设计原则：

| 字段 | 用途 | 约束 |
|------|------|------|
| 主诉 | 一句话概括核心诉求 | 尽量短，不超过 30 字 |
| 现病史 | 时间线、诱因、演变 | 只归纳原文 |
| 症状 / 伴随症状 | 阳性症状 | 数组 |
| 否认症状 | 明确否认的信息 | 未提及不等于否认 |
| 既往史 / 用药史 / 过敏史 | 病史相关 | 不详时写“未提及” |
| 生命体征 | 体温、血压、心率等 | 对象；只记录文本出现值 |
| 风险信号 | 胸痛、意识障碍、呼吸困难等 | 只记录文本明确出现项 |
| 初步分诊 | 急诊 / 普通门诊 / 随访观察 / 信息不足 | 枚举 |
| 建议就诊科室 | 科室归类 | 枚举或短文本 |
| 随访问题 | 信息不足时需要追问 | 数组 |

## 7.3 训练任务派生

复用 InstructIE 的四类任务思想，但改为医疗场景：

| 任务类型 | 建议占比 | 训练目标 |
|----------|----------|----------|
| `medical_record_extraction` | 45% | 问诊文本 -> 完整结构化记录 |
| `schema_constraint` | 20% | 只输出 schema 字段，枚举值合法 |
| `format_following` | 15% | 只输出 JSON，不输出解释和 Markdown |
| `negation_uncertainty` | 10% | 区分阳性、否认、未提及、不确定 |
| `schema_repair` | 10% | 修复字段名错误、类型错误、幻觉字段 |

推荐数据目录：

```text
data/medical_raw/
  train.jsonl
  valid.jsonl
  schema.json
data/medical_processed/
  normalized.jsonl
  filtered.jsonl
  derived_tasks.jsonl
data/medical_sft_candidate/
  train.jsonl
  valid.jsonl
  metadata.json
```

每条 SFT 样本保持当前项目的 chat-style 格式：

```json
{
  "id": "medical_record_extraction_000001",
  "source": "medical_consultation",
  "task_type": "medical_record_extraction",
  "topic_schema": "问诊记录",
  "quality_tier": "high",
  "messages": [
    {"role": "user", "content": "请将以下问诊对话转为结构化 JSON..."},
    {"role": "assistant", "content": "{\"问诊记录\": {...}}"}
  ]
}
```

## 7.4 具体提示词模板

### 7.4.1 System Prompt

```text
你是一个医疗问诊记录结构化助手。你的任务是把医生与患者的问诊对话或病情描述转写为结构化 JSON 记录。

必须遵守：
1. 只输出合法 JSON，不要输出解释、免责声明、Markdown 或代码块。
2. 只记录原文明确提到的信息，不要编造诊断、检查结果、治疗方案或药物剂量。
3. 区分“明确否认”“未提及”“不确定”。未提及的信息不要写入“否认症状”。
4. 不给出医疗建议，不替代医生诊断；“初步分诊”和“建议就诊科室”只能依据文本中的症状进行记录归类。
5. 字段名必须严格使用 schema 中给出的中文字段名。
```

### 7.4.2 医疗记录抽取 Prompt

```text
请根据给定 schema，将下面的问诊内容整理为结构化 JSON。

Schema:
["主诉","现病史","起病时间","症状","伴随症状","否认症状","既往史","用药史","过敏史","生命体征","风险信号","初步分诊","建议就诊科室","随访问题"]

要求：
- JSON 根对象固定为 "问诊记录"。
- "症状"、"伴随症状"、"否认症状"、"风险信号"、"随访问题" 使用数组。
- "生命体征" 使用对象。
- "初步分诊" 只能为 ["急诊","普通门诊","随访观察","信息不足"] 之一。
- 未提及字段不要编造；必要时写 "未提及" 或空数组。
- 只输出 JSON。

问诊内容:
{consultation_text}
```

### 7.4.3 否认与不确定专项 Prompt

```text
请从问诊内容中抽取结构化记录，特别注意区分阳性症状、明确否认症状和未提及信息。

抽取规则：
1. 患者说“没有/否认/不伴有/无”才可以进入 "否认症状"。
2. 患者说“可能/好像/不确定/记不清”应写入 "现病史"，不要当作确定事实。
3. 文本没有出现的信息不要补充。
4. 只输出合法 JSON。

Schema:
["主诉","现病史","症状","伴随症状","否认症状","既往史","用药史","过敏史","风险信号","随访问题"]

问诊内容:
{consultation_text}
```

### 7.4.4 Schema Repair Prompt

```text
下面是一份问诊结构化 JSON 草稿，可能存在字段名错误、字段类型错误、额外字段或 JSON 格式问题。
请按目标 schema 修复，保留原文可支持的信息，删除 schema 外字段，不要补充原文没有的信息。

目标 schema:
["主诉","现病史","起病时间","症状","伴随症状","否认症状","既往史","用药史","过敏史","生命体征","风险信号","初步分诊","建议就诊科室","随访问题"]

枚举约束:
- 初步分诊: ["急诊","普通门诊","随访观察","信息不足"]

错误 JSON 草稿:
{broken_json}

只输出修复后的合法 JSON。
```

## 7.5 微调全流程

### 阶段 M0：准备数据

数据来源建议：

- 已授权、已脱敏的真实问诊对话；
- 医疗客服或互联网医院问诊文本，必须确认授权与合规；
- 公开医疗对话数据集，必须检查 license；
- 用强模型生成的合成问诊样本，必须做人工或规则抽检，不能直接当高置信医学事实。

数据原始格式建议：

```json
{
  "id": "raw_000001",
  "dialogue": [
    {"role": "doctor", "text": "哪里不舒服？"},
    {"role": "patient", "text": "发热咳嗽两天..."}
  ],
  "gold_record": {
    "问诊记录": {
      "主诉": "发热咳嗽2天",
      "症状": ["发热", "咳嗽"]
    }
  }
}
```

### 阶段 M1：去标识化与标准化

已提供 `scripts/medical_01_normalize.py`：

- 合并多轮对话为统一文本；
- 统一角色名：医生 / 患者 / 家属；
- 移除姓名、电话、身份证号、住址、医院号；
- 统一温度、血压、心率等生命体征格式；
- 输出 `data/medical_processed/normalized.jsonl`。

### 阶段 M2：过滤与质量分层

已提供 `scripts/medical_02_filter.py` 与 `scripts/medical_03_quality_tier.py`：

- 剔除空文本、过短文本、无问诊信息文本；
- 剔除无法脱敏或疑似残留隐私的样本；
- 剔除 assistant 输出不是合法 JSON 的样本；
- high 质量：JSON 合法、字段覆盖完整、无隐私残留、关键字段有证据；
- medium 质量：字段少但可用；
- low 质量：噪声大，只用于分析，不进正式 SFT。

### 阶段 M3：任务派生与采样

已提供 `scripts/medical_04_derive_tasks.py` 与 `scripts/medical_05_stratified_sample.py`：

- 从 gold record 派生五类任务；
- 按科室/症状族/任务类型/复杂度分层采样；
- 重点均衡呼吸、消化、皮肤、儿科、妇产、心血管、神经、慢病随访等场景；
- 保持 high 质量样本优先。

### 阶段 M4：转写为 chat-style JSONL

已提供 `scripts/medical_06_to_chat_jsonl.py`：

- 输出 `data/medical_sft_candidate/train.jsonl`；
- 输出 `data/medical_sft_candidate/valid.jsonl`；
- 生成 `metadata.json` 和 spotcheck 样本；
- 对 assistant 内容执行 `json.loads` 校验，目标 100% 合法。

使用仓库内的去标识化 mock 数据可直接跑通六步 smoke：

```bash
python3 scripts/medical_01_normalize.py
python3 scripts/medical_02_filter.py
python3 scripts/medical_03_quality_tier.py
python3 scripts/medical_04_derive_tasks.py
python3 scripts/medical_05_stratified_sample.py
python3 scripts/medical_06_to_chat_jsonl.py
```

当前 smoke 输入为 `data/medical_raw/smoke.jsonl`。真实项目接入时替换该目录下的授权、去标识化问诊数据，再重新运行六步 pipeline 即可。`data/medical_processed/`、`data/medical_sft_candidate/` 和 `reports/medical/` 均为可再生成产物。

### 阶段 M5：Qwen LoRA 训练

已提供配置：

- `configs/qwen_lora_medical_structured_smoke.json`
- `configs/qwen_lora_medical_structured.json`

正式配置核心内容如下：

### 阶段 M6：评测与部署验证

已提供 `eval/prompts_medical_consultation.json`，可复用现有离线评测：

```bash
python scripts/run_instructie_eval.py \
  --eval-file eval/prompts_medical_consultation.json \
  --skip-microlm \
  --out-dir results/medical_consultation_eval
```

vLLM 部署后：

```bash
python scripts/check_structured_stability.py \
  --eval-file eval/prompts_medical_consultation.json \
  --output-dir results/medical_consultation_vllm
```

医疗专项指标建议：

| 指标 | 含义 |
|------|------|
| Parse% | JSON 可解析率 |
| Strict% | 必填字段、额外字段、枚举约束是否通过 |
| Slot-F1 | 字段值与 gold 的精确/包含匹配 |
| Negation-Accuracy | 否认症状是否被错误放入阳性症状 |
| No-Invention% | 是否避免编造未提及信息 |
| Safety-Boundary% | 是否避免输出诊断/处方/建议性文本 |

当前 `scripts/run_instructie_eval.py` 和 `scripts/check_structured_stability.py` 已补充医疗字段 alias，并在 prompt rules 请求时输出附加诊断：

- `medical_no_invention_rate`
- `medical_negation_accuracy_rate`
- `medical_safety_boundary_rate`

这些是医疗专项辅助指标，不替代原有 Parse% / Strict% / Alias-Strict% 主指标。

## 7.7 改进路线

1. **短期**：先加入医疗 eval prompt，直接评估 base Qwen 与现有 LoRA 的零样本能力。
2. **中期**：用少量脱敏/合成数据跑 smoke LoRA，验证 JSON 稳定性、否认识别、枚举分诊。
3. **长期**：接入真实授权数据、人工抽检、医疗安全评测和线上审计日志。

医疗场景的关键收益是：结构化质量可量化、输出边界清晰、容易接入电子病历或质控系统；关键风险是：数据合规、幻觉、否认误判和过度医疗建议。因此微调目标必须始终限定为“记录抽取”，而不是“医学决策”。
