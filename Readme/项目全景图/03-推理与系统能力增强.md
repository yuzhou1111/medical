# 三、推理与系统实现

> 这一章讲的是：训练出来的权重，如何变成真正能用的系统。

---

## 3.1 文本生成与自回归推理流程

### 这一小节要解决什么问题

`generate_text.py` 是单轮推理的统一入口——它需要解决的是"给定一个 checkpoint 和一段输入文本，如何让模型生成输出"。核心挑战是 prompt 格式组装（保证与训练一致）和自回归 decode 循环的正确实现。

### 核心设计 / 核心流程

**入口脚本支持两种输入模式：**

| 模式 | 参数 | 适用场景 |
|------|------|----------|
| 纯文本续写 | `--prompt "..."` | 观察 pretrain checkpoint 的语言建模能力 |
| 对话生成 | `--conversations-json '...'` 或 `--conversations-path file.json` | SFT / LoRA checkpoint 的指令跟随能力 |

其他运行时参数：`--dtype`（float32/float16/bfloat16）、`--temperature`、`--top-p`、`--eos-token`、`--max-new-tokens`、`--seed`、`--show-token-ids`、`--print-new-text-only`、`--config-path`

**Prompt 组装**由 `prompting.py` 的 `resolve_generation_prompt()` 负责：

- **纯文本模式**：直接 tokenizer.encode → 截断 → 送入 generate
- **对话模式**：先调用 `build_generation_prompt()`（来自 sft.py），在末尾补上 assistant 标记，确保推理时的 prompt 格式与 SFT 训练时的格式严格一致

格式一致性是这里的关键约束。SFT 训练时 assistant 回复以特定标记开头，推理时如果遗漏这个标记，模型看到的第一个 token 分布就和训练时不匹配，导致生成质量下降。

**自回归生成循环**封装在 `TransformerLM.generate()` 中：

```
输入: prompt_ids (shape: [seq_len])
     ↓
[Phase 1: Prefill]
  model.forward(prompt_ids, use_cache=True, start_pos=0)
  → 完整 prompt 通过所有层，每层缓存 K/V
  → 取最后一个位置的 logits
     ↓
[Phase 2: Decode Loop]  (重复 max_new_tokens 次)
  ① temperature 缩放 logits（若 temperature ≠ 1.0）
  ② top-p nucleus sampling（若 top_p < 1.0）
     - 按 logit 降序排列
     - 累积概率，mask 掉超过阈值的 token
     - "右移一位"技巧：保留第一个越界 token
  ③ softmax + multinomial 采样 → next_token
  ④ 检查 EOS：若命中则终止
  ⑤ model.forward([next_token], kv_cache=cached, start_pos=cur_pos)
     → 只计算 1 个 token 的前向传播
     → 新 K/V 拼接到每层缓存末尾
  ⑥ cur_pos += 1
     ↓
输出: generated_ids (prompt + new_tokens)
```

top-p 实现细节：对 logits 降序排列后计算累积概率分布，将累积概率首次超过阈值之后的 token 全部 mask 为 -inf。但使用"右移一位"技巧——第一个导致超标的 token 仍然保留（否则极端情况下可能 mask 掉所有 token）。

### 关键结果 / 数据

这一节的核心产出是可用的单轮推理管线。关键设计决策：
- **每轮只取最后一个位置的 logits**：prefill 阶段完整序列通过所有层后，只有最后一个位置的 logits 用于采样——因为自回归模型只需要预测下一个 token
- **temperature 缩放**：控制输出随机性，temperature > 1.0 更多样，< 1.0 更确定
- **top-p nucleus sampling**：从累积概率超过阈值 p 的候选集中采样，平衡质量和多样性
- **EOS 终止判断**：遇到 EOS token 或达到 max_new_tokens 时停止生成

### 这一节的结论

文本生成管线的设计核心是**训练-推理格式一致性**。SFT 阶段定义的数据协议（role markers + EOS + loss mask）必须在推理侧精确复现，任何偏差都会静默地降低生成质量。KV Cache 的集成让 decode 阶段的计算量从 O(n²) 降为 O(n)，使长序列生成成为可能。

## 3.2 KV Cache 优化与 Benchmark

### 这一小节要解决什么问题

标准自回归解码中，每生成一个新 token 都要将完整序列重新送入模型。对于长度为 n 的序列生成 m 个新 token，总计算量为 O(m·n²) —— 序列越长，冗余计算越多。KV Cache 的目标是消除这种冗余。

### 核心设计 / 核心流程

**KV Cache 的核心思想：** Prefill 阶段每层的 Key 和 Value 只算一次并缓存起来，Decode 阶段每次只把新 token 的 K/V 拼接到缓存末尾。这样每个 decode step 的计算量从 O(n²) 降为 O(n)（只需新 query 与历史 K/V 做注意力）。

**在 TransformerLM 中的实现**通过三个参数控制：

| 参数 | 类型 | 作用 |
|------|------|------|
| `use_cache` | bool | 是否启用 KV Cache |
| `kv_cache` | `KVCache \| None` | 已有的缓存（首调用为 None）|
| `start_pos` | int | 当前 token 在完整序列中的位置（用于 RoPE 计算）|

`KVCache` 本身是一个轻量容器——每层维护一个 K 张量和一个 V 张量，初始化为 None。Prefill 时创建并填充，Decode 时通过 `torch.cat` 沿 sequence dimension 拼接。

Decode 阶段的两个优化：
- **跳过 causal mask**：只有 1 个 query token，无需下三角掩码
- **注意力 score 计算 float32**：即使模型是 FP16/FP32，attention 内部也 cast 到 float32 保证数值稳定性

**Benchmark 设计：**

测试矩阵：5 种 prompt 长度（16 / 32 / 64 / 128 / 256）× 4 种生成长度（32 / 64 / 128 / 256）= **20 组配置**

每组执行流程：预热 1 次 + 正式测量 3 次 → 取平均。固定变量：同一模型、同一 dtype、同一 device、同一随机种子。

### 关键结果 / 数据

**核心结果**（CPU, float32 环境）：

| 指标 | 数值 |
|------|------|
| 平均加速比 | **3.86x** |
| 最大加速比 | **9.08x**（prompt=256, gen=256）|
| Cache decode 吞吐 | **~100 tok/s**（恒定）|
| No-cache 吞吐 | 随序列长度**二次衰减** |

**关键发现：**

- Cache 方案的 decode 吞吐恒定在 ~100 tok/s，不随已生成序列长度增加而下降——因为每步的计算量是固定的（1 个 token 的 query × 缓存的 K/V）
- No-cache 方案的吞吐随序列总长近似二次衰减——每步都要重算更长序列的注意力
- 加速比与序列总长正相关：prompt 越长、生成长度越大，KV Cache 的优势越明显

### 这一节的结论

平均加速 3.86x、最大 9.08x 的数据证明了 KV Cache 在自回归推理中的必要性。更重要的发现是 **cache 方案实现了恒定吞吐（~100 tok/s）**，这意味着无论生成了多长的序列，每一步 decode 的延迟都是可预测的——这对在线服务的 SLA 设计有直接价值。

> [!note] 与 vLLM benchmark 的关系
> 这里的 KV Cache benchmark 测量的是**算法层面**的加速比（no-cache vs cache），在 CPU float32 环境下运行。第五大主题中的 vLLM benchmark 测量的是**系统层面**的服务性能（TTFT / 吞吐 / P95 延迟），在 GPU 上运行。两者互补：前者证明优化思路有效，后者证明部署后实际表现达标。

## 3.3 交互式对话系统 chat.py

### 这一小节要解决什么问题

`chat.py` 将模型从"单次生成脚本"升级为**可交互的多轮对话系统**。它解决的问题不是"模型能不能生成文本"，而是"用户能不能直观地感受和对比不同版本模型的差异"——这对项目演示和调试都至关重要。

### 核心设计 / 核心流程

**多模型支持**——同一个 chat.py 可以加载三种模型状态：

| 加载方式 | 行为 |
|----------|------|
| Pretrain checkpoint | 基础续写模式，适合观察语言建模能力 |
| SFT baseline checkpoint | 全参微调后的对话助手 |
| SFT LoRA checkpoint + adaptor | 先 merge_lora 再推理，LoRA 微调版 |

三者的差异仅在 checkpoint 文件和可选的 LoRA adaptor，模型架构完全相同。LoRA 加载时自动执行 merge 操作，对上层调用透明。

**REPL 命令：**

| 命令 | 功能 |
|------|------|
| `/temp <value>` | 运行时切换 temperature |
| `/topp <value>` | 运行时切换 top-p |
| `/system <prompt>` | 设置或更换 system prompt |
| `/clear` | 清空对话历史，重新开始 |
| `/history` | 查看当前对话历史（轮次统计）|
| `/save [path]` | 保存会话日志到 JSONL 文件（含时间戳）|
| `/quit` | 退出 |

temperature 和 top_p 的运行时切换不需要重启会话——下一次生成立即生效。这方便在同一轮对话中快速对比不同采样参数的效果。

**多轮对话机制：**

1. 用户输入追加到 conversations 列表
2. 调用 `build_generation_prompt()` 渲染完整对话历史为线性文本
3. tokenizer 编码后检查是否超出 context_length（512）
4. 若超长，**自动裁剪最早的历史轮次**，保留最近的对话上下文
5. 送入 model.generate() 生成回复
6. 回复追加到 conversations，回到步骤 1

上下文窗口自动截断是一个重要的工程细节——没有它，多轮对话会在第 4-5 轮后因超出 context_length 而 crash 或退化。

**内部架构：**

```
CLI (argparse: checkpoint / lora / config / vocab)
    → load_chat_model()  加载 tokenizer + TransformerLM + state_dict + 可选 merge_lora
    → ChatSession
        conversations 列表
        _render_prompt()  ROLE_MARKERS 格式化 + 条件性追加 EOS
        chat()  encode → clip → generate → decode → sanitize → 存储
    → repl()  输入分发 / 命令处理 / 计时 / 日志 / 异常捕获
```

### 关键结果 / 数据

测试验证覆盖 7 个场景：SFT baseline 单轮/多轮、LoRA 单轮/多轮、Pretrain 单轮/多轮、REPL 全命令、会话日志、含 surrogate 字符的多轮场景——全部通过。

**修复的关键 Bug：**

**Bug 1 — EOS 标记缺失**：`_render_prompt()` 在 assistant 消息后缺少 EOS 标记，导致推理时的 prompt 格式与 SFT 训练时不一致。训练数据中每条 assistant 回复都以 EOS 结尾，推理时如果漏掉这个标记，模型学到的"看到 assistant 标记后输出一段文字然后停"的模式就被破坏了。

**Bug 2 — Token ID 越界**：多轮对话第二轮时崩溃。原因是 tokenizer 词表大小为 6401（含 special token），但模型 embedding 层的 vocab_size 只有 6400。EOS token 的 ID=6400 超出了 embedding 的索引范围。修复方案：编码后裁剪越界 token ID，且仅在 eos_token_id 有效（< vocab_size）时才追加 EOS 标记。

**Bug 3 — Unicode Surrogate 字符崩溃**：小模型生成的某些 token 序列解码后产生 Unicode surrogate 字符（U+D800–U+DFFF），存入对话历史后下一轮 `.encode("utf-8")` 时抛出 `UnicodeEncodeError`。新增 `_remove_surrogates()` 工具函数，每轮回复存入历史前自动清理。这类 bug 的特征是只在多轮对话的第二轮及之后触发——问题出在"历史的累积"而非单次生成。

**当前限制：**

| 限制 | 原因 | 影响 |
|------|------|------|
| EOS 自动停止不生效 | tokenizer 词表(6401) > model embedding(6400) | 需靠 max_new_tokens 截断 |
| 生成质量有限 | 31.7M 参数 + 中文语料规模有限 | 输出可能出现 repetition loop |
| 不支持流式输出 | 完整生成后一次性返回 | 长回复需等待全部完成 |

### 这一节的结论

chat.py 完成了从"脚本推理"到"可交互系统"的跨越。三个 bug 的修复过程揭示了一个共同模式：**累积型 bug（历史状态逐步恶化）比一次性 bug 更难排查**——因为单次运行正常，问题只在多轮累积后才暴露。对任何涉及"历史""缓存""累积"的代码都要特别警惕。
