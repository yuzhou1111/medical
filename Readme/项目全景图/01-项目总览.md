# 一、项目总览


---

## 1.1 项目背景与目标

MicroLM 的起点是一个朴素的问题：**从零搭建一个能训练、能微调、能推理的完整 LLM 链路，需要哪些环节？**

大多数 LLM 学习路径停留在调用 HuggingFace Trainer 或跑通一个微调脚本。这个项目的目标更底层——把 tokenizer 训练、语料处理、pretrain、SFT、LoRA、推理优化、评测、部署每一个环节都亲手实现一遍，形成可验证、可对比、可演示的完整闭环。

**项目的起点是什么：** 从零搭建一条完整的 LLM 训练/微调/推理链路，覆盖 MLOps 全流程。

**为什么要做"自研 + 迁移"两条线：** 自研 MicroLM 链路（31.7M 参数）证明的是"从零搭建"的能力——tokenizer、模型结构、训练循环全部手写。Qwen 迁移链路（1.55B 参数）证明的是"在开源生态中适配"的能力——用 HF/PEFT/vLLM 等工业工具完成同样的事情。两者共享设计模式（配置组织、日志格式、训练循环结构），但技术栈完全不同。

**为什么阶段 6 改为聚焦结构化抽取：** 项目执行过程中经历了一次关键的方向聚焦。阶段 0-5 完成后，项目具备了通用聊天能力；但在进入阶段 6 时做了一个 deliberate decision：**不再做泛化型中文助手增强，而是聚焦于 InstructIE 驱动的结构化信息抽取与 Schema 约束输出**。原因很实际——这类任务有明确的硬指标（JSON 可解析率、字段准确率、格式遵循率），LoRA 擅长的"行为约束 + 格式塑形"在这里能发挥最大价值，且直接服务于后续 Agent / tool call / function calling 场景。

> [!tip] 两条链路的分工逻辑
> 自研线不是 Qwen 线的"前置依赖"，而是并行的能力验证。复用的设计模式：配置文件组织方式、JSONL 日志格式、训练循环结构、报告模板。重新适配的部分：tokenizer 接口（BPE → HF AutoTokenizer）、Chat 模板（自定义 markers → ChatML）、LoRA 框架（自研 lora.py → PEFT）、Loss mask 构造方式（标记定位 → prefix 对比法）。

## 1.2 项目整体架构

### 全局数据流

```
┌─ 主线 A+B：自研 MicroLM 链路 ─────────────────────────────┐
│                                                            │
│  原始语料 (~141万条)                                        │
│    → 清洗切分 → BPE tokenizer (vocab=6400)                 │
│    → tokenization (.npy memmap)                            │
│    → Pretrain (8层 Transformer, RoPE+SwiGLU, 31.7M参数)    │
│    → SFT 全参微调 / LoRA 微调 (0.83% 可训练参数)            │
│    → 推理 (KV Cache 加速) → chat REPL / 评测 / benchmark   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌─ 主线 C+D：Qwen 迁移与部署链路 ───────────────────────────┐
│                                                            │
│  InstructIE 原始数据 (171K 条, 12主题)                      │
│    → 6步数据 pipeline (标准化→过滤→分层→派生→采样→转写)      │
│    → SFT 数据集 (28.5K train + 1.5K valid)                  │
│    → Qwen2.5-1.5B LoRA 微调 (0.14% 可训练参数)              │
│    → 结构化评测 (4模型 × 40prompt × 4指标)                   │
│    → LoRA merge 导出 → vLLM 部署 → smoke/benchmark/稳定性   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 自研链路：MicroLM（主线 A → B）

这条线从原始字节开始搭建，核心组件全部自研：

| 环节        | 实现                                                       | 关键产出                   |
| --------- | -------------------------------------------------------- | ---------------------- |
| Tokenizer | byte-level BPE，GPT-2 风格预分词                               | vocab=6400             |
| 模型        | TransformerLM：8 层 RoPE + SwiGLU + RMSNorm pre-norm       | 31.7M 参数               |
| Pretrain  | 自实现 AdamW + cosine scheduler + gradient clipping         | `ckpt_final.pt`        |
| SFT       | conversations → chat prompt → assistant-only masked loss | baseline val_loss=2.33 |
| LoRA      | 自研 LoRALinear（einsum，支持 merge/unmerge）                   | 0.83% 参数，99.7% 存储节省    |
| 推理        | KV Cache 增量解码 + top-p sampling + temperature scaling     | 平均加速 3.86x             |

SFT 阶段的数据协议是整条链路的关键设计决策：对话数据经过 `normalize → render → encode → build_loss_labels` 四步处理，只有 assistant 回答区间参与 loss 计算，其余位置置为 -100。这保证了 pretrain 权重在 SFT 阶段不会被 prompt 部分的梯度污染。

### 迁移链路：Qwen 结构化输出（主线 C → D）

这条线将整套方法论迁移到开源生态，同时将能力方向收窄到结构化抽取：

| 环节 | 工具/方法 | 关键产出 |
|------|-----------|----------|
| 数据构建 | 6 步 pipeline（阈值集中在 `conf.py`） | 28.5K 条，4 类任务，JSON 合法性 100% |
| 微调 | HF AutoModel + PEFT LoRA + ChatML 模板 | val_loss 0.40→0.15（降幅 61.6%） |
| 评测 | 3 组 × 40 条 prompt × 4 项自动检测指标 | 推荐 qwen_lora 部署 |
| 部署 | PEFT merge_and_unload → vLLM OpenAI 兼容 API | smoke 5/5，stability Parse% 100% |

数据 pipeline 是这条线的工程核心。从 InstructIE 的 17 万条原始数据出发，经过标准化（字段对齐）、两层过滤（硬过滤 3,585 条 + per-topic P99 软过滤 4,257 条）、质量三档分层、四类任务派生（ie_extraction / text_to_json / format_following / schema_repair）、按 task_type+topic+quality 分层采样，最终产出 30,000 条候选集（28.5K train + 1.5K valid），全量 JSON 校验通过。

### 两线关系

自研线不是 Qwen 线的"前置依赖"，而是**并行的能力验证**：

- 复用的设计模式：配置文件组织方式、JSONL 日志格式、训练循环结构、报告模板
- 重新适配的部分：tokenizer 接口（BPE → HF AutoTokenizer）、Chat 模板（自定义 markers → ChatML）、LoRA 框架（自研 lora.py → PEFT）、Loss mask 构造方式（标记定位 → prefix 对比法）
- 评测阶段的交汇：`run_instructie_eval.py` 同时对 qwen_base / qwen_lora / microlm_sft / microlm_lora 四个模型做统一对比

## 1.3 最终成果与核心亮点




### 核心量化成果

**自研 MicroLM 链路：**

| 指标              | 数值                                              |
| --------------- | ----------------------------------------------- |
| 模型参数量           | 31,729,152（~31.7M）                              |
| LoRA 可训练参数占比    | 0.83%（262K / 31.7M）                             |
| LoRA adaptor 存储 | 1.0 MB（vs 全参 377 MB，节省 99.7%）                   |
| SFT val_loss    | baseline 2.3265 / LoRA 2.5377（差距 9%）            |
| KV Cache 平均加速   | **3.86x**（最大 9.08x）                             |
| SFT 生成质量提升      | 平均评分 +81%（pretrain 1.13 → baseline 2.04 / 满分 5） |

**Qwen 迁移链路：**

| 指标               | 数值                                                  |
| ---------------- | --------------------------------------------------- |
| 基座模型             | Qwen2.5-1.5B-Instruct（1.55B 参数）                     |
| LoRA 可训练参数占比     | 0.14%（2.18M / 1.55B）                                |
| LoRA adaptor 存储  | 8.3 MB（vs 基座 2,944 MB，节省 99.7%）                     |
| SFT 数据集规模        | 28,500 train + 1,500 valid（InstructIE 单源）           |
| val_loss 降幅      | 0.3999 → **0.1534**（61.6%，2000 步）                   |
| Alias-Strict 命中率 | **15.0%**（为 base 的 **2 倍**）                         |
| 结构化行为塑形          | 实体做 key 率 95.0%（+37.5pp），字段名重叠率 49.0%（+44.5pp）      |
| vLLM 部署验证        | smoke 5/5 通过，benchmark 5 组 0 错误，稳定性 Parse% **100%** |

### 核心亮点

> [!note] 从零到服务的完整闭环
> 项目覆盖了 MLOps 的每一个环节：数据获取与清洗 → tokenizer 训练 → 模型预训练 → 指令微调 → 参数高效微调 → 评测体系构建 → 服务化部署 → 性能量化与稳定性验证。这不是一组散落的实验脚本，而是一条可以从原始数据一路跑到线上 API 的完整管线。

**双轨并行验证同一套方法论。** 自研链路用纯 PyTorch + einsum 实现所有组件（包括自定义 Linear 层和 LoRALinear），Qwen 链路用 HF/PEFT/vLLM 等工业工具栈完成相同流程。两者的配置组织、日志格式、训练循环结构和报告模板高度一致，证明了方法论的可迁移性。

**结构化评测体系替代主观打分。** 在 Qwen 线上完全放弃了人工评分，改用 4 项自动检测指标（JSON 可解析率 / 缺字段率 / 幻觉字段率 / 严格 Schema 命中率）对 4 个模型做统一量化对比，并附加 alias 归一化和结构化质量分析。这让"模型变好了"从一个感受变成了可复现的数字。

**数据处理 pipeline 工程化。** 6 步数据处理的阈值集中配置在 `conf.py`，每一步产出独立的统计报告（JSON 格式），硬过滤和软过滤（per-topic P99 分位数）两层把关，质量三档分层确保采样只取 high 质量样本，最终全量 JSON 合法性校验 100% 通过。
